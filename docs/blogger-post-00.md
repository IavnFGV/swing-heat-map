## Что я сейчас строю

Это прототип heatmap для визуализации рыночных данных, написанный на обычном Java Swing.

Сейчас он выглядит примерно так:

> Здесь вставить `artifacts/current/01-current-ui.png`.

В приложении уже есть:

- heatmap;
- стакан;
- исторические сделки;
- CVD и объём;
- несколько режимов хранения данных;
- встроенные метрики.

Важное уточнение: это не торговый продукт и не попытка собрать Bookmap дома из палок, Swing и уверенности в себе.

Это инженерный эксперимент.

Мне интересно проверить:

- насколько далеко вообще можно зайти с plain Java Swing;
- при какой нагрузке интерфейс начинает грустить;
- что на самом деле становится bottleneck;
- помогает ли очередная «очевидная оптимизация» или просто делает код менее понятным.

## Начиналось всё подозрительно просто

Первая версия помещалась в один `JPanel`.

```java
private static final int PRICE_LEVELS = 200;
private static final int TIME_BUCKETS = 800;

private final int[][] heatmap =
        new int[PRICE_LEVELS][TIME_BUCKETS];
```

Двумерный массив, `Swing Timer` и немного рисования.

Каждые 100 миллисекунд содержимое матрицы сдвигалось влево, справа появлялась новая колонка, а `paintComponent` снова проходил по данным и рисовал непустые ячейки.

И оно работало.

Это вообще опасный момент в любом прототипе. Когда первая версия работает, мозг немедленно выдаёт сертификат: «Архитектура подтверждена. Можно добавлять ещё семь функций».

Я добавил.

Появился synthetic order book. Обновление стало происходить примерно 60 раз в секунду. За один tick начали применяться тысячи событий.

И вопрос постепенно изменился.

Сначала он звучал так:

> Можно ли нарисовать такую визуализацию на Swing?

Теперь он звучит так:

> Что именно я заставляю компьютер делать ради одной новой колонки и успевает ли он закончить это до следующего кадра?

## Что хочу проверить

Я собираюсь пройти по истории Git, а не сочинять её задним числом:

1. Запустить первый рабочий коммит.
2. Найти момент, когда появилась содержательная нагрузка.
3. Записать подозреваемых до профилирования.
4. Измерить генерацию данных, копирование и отрисовку.
5. Проверить, что действительно мешало.
6. Сравнить состояние до и после изменения.

Подозреваемые уже есть:

- `System.arraycopy` для каждой строки heatmap;
- полный проход по матрице при каждом repaint;
- создание объектов во время рисования;
- генерация данных на EDT;
- моё традиционное «давайте просто увеличим число ещё в десять раз».

Но пока это только список подозреваемых. Назначать виновного по внешнему виду кода было бы слишком удобно.
