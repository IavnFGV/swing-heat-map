## 1. Рабочая идея видео

Рабочий заголовок:

> How far can plain Java Swing be pushed for real-time visualization?

Более узкий заголовок первого выпуска:

> Почему первая версия моей Swing heatmap начала тормозить

Главная история:

> Я сделал простую heatmap на Java Swing. Первая версия появилась быстро и работала. Затем я начал увеличивать объём данных и развивать интерфейс. Вместо рассказа о готовом коде вернёмся к первому коммиту, воспроизведём его ограничения и посмотрим, какие измерения привели к следующей реализации.

Это не урок по созданию торгового приложения и не обзор готового продукта. Это инженерное расследование.

## 2. Правила первого видео

- Продолжительность: ориентир 5–8 минут.
- Без камеры.
- Сначала записывается экран, затем текст и голос.
- Один главный вопрос.
- Один подтверждённый технический вывод.
- Не пытаться показать всю историю репозитория.
- Не скрывать, если первоначальная гипотеза оказалась неверной.
- Не использовать неподтверждённые показатели производительности.
- Современная версия используется как визуальная затравка, а не как benchmark.

### Сцена 3. Объяснить устройство первой версии

Ориентир: `0:40–1:30`.

Видеоряд:

- `HeatmapPanel.java`;
- последовательно подсветить только четыре части:
  1. `int[200][800]`;
  2. `Timer(100, ...)`;
  3. сдвиг строк через `System.arraycopy`;
  4. двойной цикл в `paintComponent`.

Факты из кода:

```text
PRICE_LEVELS = 200
TIME_BUCKETS = 800
Timer interval = 100 ms
100 synthetic updates per tick
```

Черновой текст:

> Данные лежат в матрице 200 на 800. Каждые 100 миллисекунд новая колонка добавляется справа. Чтобы освободить для неё место, каждая строка целиком сдвигается влево. Затем `paintComponent` снова проходит по всей матрице и рисует непустые ячейки.

Дополнительный визуальный beat:

```text
200 × 799 = 159 800 copied int / tick
200 × 800 = 160 000 checked cells / paint
```

Фраза для голоса:

> Название метода и цена операции иногда живут в разных районах города. Добавление одной колонки здесь означает сдвиг почти всей истории.

Важно:

- пока не называть `System.arraycopy` причиной тормозов;
- говорить только о том, какую работу выполняет программа.

Статус:

- [x] нужный код найден;
- [ ] подготовить удобный масштаб редактора;
- [ ] записать короткий проход по коду.

### Сцена 4. Увеличить нагрузку

Ориентир: `1:30–2:30`.

Видеоряд:

- baseline с исходными параметрами;
- затем изменённые параметры нагрузки;
- видимая потеря плавности или другие измеримые ограничения.

Черновой текст:

> На исходных размерах всё выглядит нормально. Поэтому следующий вопрос — не «работает ли Swing», а «при какой работе эта реализация перестаёт укладываться во время кадра».

Нужно определить:

- какой один параметр увеличивать первым;
- при каких значениях проблема хорошо видна;
- нужен ли счётчик FPS в экспериментальной ветке;
- как не смешать генерацию данных и rendering cost.

Исторические кандидаты:

```text
b0658a5 — первый экран: Timer 100 ms, 100 synthetic updates
45d73e8 — Timer 16 ms, synthetic order book, 1000 events/tick
c5c2073 — те же основные параметры вынесены в MarketConfig и показаны на UI
```

Рабочая драматургия:

> Первый коммит нужен, чтобы показать происхождение идеи. Затем проект делает небольшой, но важный шаг: вместо случайных пятен появляется synthetic order book, частота обновления вырастает примерно до 60 раз в секунду, а на каждом tick применяется 1000 событий. Простая архитектура остаётся прежней — и теперь у неё появляется содержательная нагрузка.

Оценка запрошенной работы для подписи:

```text
~9.6M copied int/sec
~9.6M checked cells/sec
up to 60K configured events/sec
```

Обязательно подписать:

```text
Estimated requested work — not measured throughput
```

Статус:

- [ ] сценарий нагрузки не выбран;
- [ ] проблема не воспроизведена;
- [ ] метрики не добавлены.

### Сцена 5. Сформулировать гипотезы

Ориентир: `2:30–3:10`.

Видеоряд:

- короткий список рядом с работающим приложением;
- фрагменты соответствующего кода.

Черновой текст:

> Здесь есть сразу несколько подозреваемых: копирование строк, полный проход при каждом repaint, создание `Color`, работа генерации на EDT и сама частота перерисовки. По коду можно придумать красивое объяснение, но сначала это нужно измерить.

Возможные гипотезы:

- `System.arraycopy`;
- полный проход `PRICE_LEVELS × TIME_BUCKETS`;
- `new Color(...)` внутри render loop;
- генерация данных внутри Swing Timer на EDT;
- слишком частый или слишком тяжёлый repaint.

Статус:

- [x] предварительные гипотезы видны из кода;
