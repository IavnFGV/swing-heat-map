В прошлой записи я показал, во что постепенно превратился мой эксперимент с Java Swing.

Теперь сделаем несколько шагов назад.

Есть один JPanel.

Перед тем как оптимизировать код, полезно посчитать, какую работу мы вообще выполняем.

В этой записи я хочу сделать именно это:

  • взять первую рабочую версию;
  • найти операции, которые выполняются каждый tick;
  • прикинуть их стоимость в секунду;

Первый рабочий коммит

Вся heatmap хранилась в обычной матрице:

private static final int PRICE_LEVELS = 200;
private static final int TIME_BUCKETS = 800;

private final int[][] heatmap =
        new int[PRICE_LEVELS][TIME_BUCKETS];

То есть:

  • 200 ценовых уровней;
  • 800 временных колонок;
  • 160 000 ячеек.

Главный цикл - просто таймер

Timer timer = new Timer(100, e -> {
    generateFakeData();
    repaint();
});

Каждые 100 миллисекунд:

  1. старые данные сдвигались влево;
  2. справа создавалась новая колонка;
  3. Swing получал просьбу перерисовать панель.

Слово «просьбу» здесь стоит воспринимать буквально. repaint() не рисует немедленно, а ставит перерисовку в очередь EDT. Swing сам разберется.

Один маленький сдвиг

Для добавления новой колонки каждая строка массива сдвигалась на одну позицию:

for (int y = 0; y < PRICE_LEVELS; y++) {
    System.arraycopy(
            heatmap[y],
            1,
            heatmap[y],
            0,
            TIME_BUCKETS - 1
    );

    heatmap[y][TIME_BUCKETS - 1] = 0;
}

System.arraycopy  JVM выполянет очень хорошо.

Посчитаем операции:

200 строк × 799 элементов = 159 800 скопированных int за tick

При интервале 100 мс получается до десяти таких обновлений в секунду:

примерно 1,6 млн перемещений int в секунду

Это к тому - «добавить одну колонку» физически означает «передвинуть всю историю».

paintComponent

После обновления данных paintComponent снова проходил по всей матрице:

for (int y = 0; y < PRICE_LEVELS; y++) {
    for (int x = 0; x < TIME_BUCKETS; x++) {
        int volume = heatmap[y][x];

        if (volume == 0) {
            continue;
        }

        int intensity = Math.min(255, volume * 4);
        g.setColor(new Color(intensity, intensity / 2, 0));
        g.fillRect(...);
    }
}

Ещё 160 000 проверок на одну отрисовку.

Не каждая ячейка превращалась в прямоугольник — пустые значения пропускались. Но проверить значение всё равно нужно.

Если представить десять полных отрисовок в секунду, получим ещё примерно 1,6 млн проверок ячеек в секунду.

Для современного процессора эти цифры вполне подъемные. Поэтому первая версия и работала нормально.


На скриншоте видно главное: Timer, generateFakeData(), repaint() и тот самый сдвиг строк через System.arraycopy.

Небольшая магия: цвет и координаты

Дальше надо  наши данные вывести н аэкран - и здесь уже без нужно перейти в модель - которую нам дает Swing для работы с отображением:

double cellW = w / (double) TIME_BUCKETS;
double cellH = h / (double) PRICE_LEVELS;

int intensity = Math.min(255, volume * 4);
g.setColor(new Color(intensity, intensity / 2, 0));

int px = (int) (x * cellW);
int py = (int) ((PRICE_LEVELS - 1 - y) * cellH);

g.fillRect(
        px,
        py,
        Math.max(1, (int) cellW + 1),
        Math.max(1, (int) cellH + 1)
);

Здесь три простые идеи.

Первая: размер одной ячейки на экране.

double cellW = w / (double) TIME_BUCKETS;
double cellH = h / (double) PRICE_LEVELS;

Если у нас 800 временных колонок, а окно шириной 1200 пикселей, одна колонка занимает примерно 1.5 пикселя. Если 200 ценовых уровней и окно высотой 700 пикселей, один уровень занимает примерно 3.5 пикселя.

То есть массив не рисуется “как есть”. Каждая ячейка массива сначала переводится в прямоугольник на экране. И важно - уже здесь мы сходу получаем масштабирование -  при изменеиии размера копонента мы получим изменение и данных!

Вторая идея: объём превращается в цвет.

int intensity = Math.min(255, volume * 4);
g.setColor(new Color(intensity, intensity / 2, 0));

volume — это число в ячейке heatmap. Чем оно больше, тем ярче должен быть квадратик.

volume * 4 просто усиливает эффект, чтобы маленькие значения не были почти чёрными. А Math.min(255, ...) ставит потолок, потому что RGB-канал не может быть больше 255.

Дальше цвет собирается так:

red   = intensity
green = intensity / 2
blue  = 0

Получается оранжевый: чем больше объём, тем ярче “жар”.

Третья идея: координаты массива переводятся в координаты Swing.

int px = (int) (x * cellW);
int py = (int) ((PRICE_LEVELS - 1 - y) * cellH);

С x всё почти честно:

x = 0   → левый край
x = 799 → правый край

А вот с y есть ловушка. В массиве y - это ценовой уровень. Логически хочется, чтобы большие цены были выше. Но в Swing координата y = 0 находится сверху, а дальше y растёт вниз.

Поэтому появляется переворот:

PRICE_LEVELS - 1 - y

Без него график был бы нарисован вверх ногами. 

И последняя маленькая страховка:

Math.max(1, (int) cellW + 1)
Math.max(1, (int) cellH + 1)

Она гарантирует, что прямоугольник будет хотя бы 1 × 1 пиксель. Если окно маленькое или колонок слишком много, cellW может стать меньше одного пикселя. 

В итоге вся “магия” сводится к простому конвейеру:

heatmap[y][x]
→ volume
→ яркость оранжевого
→ x/y массива
→ px/py на экране
→ fillRect(...)

На этом месте первую версию уже можно считать разобранной.

Для первого шага это почти идеальный вариант: один JPanel, один двумерный массив, один Timer, один метод генерации и один метод рисования.

Плюс она честно показывает базовую механику:

  • время - это столбцы;
  • цена - это строки;
  • объём - это яркость;
  • движение времени - это сдвиг массива влево;
  • картинка на экране - это перевод ячеек массива в прямоугольники Swing.

Итого: сколько работы делает один tick

Теперь можно собрать всё в одну табличку. Не как настоящий benchmark, а как оценку того, сколько работы мы сами попросили у программы.

На каждый tick таймера, то есть примерно раз в 100 мс, происходит генерация новой колонки:

PRICE_LEVELS = 200
TIME_BUCKETS = 800

сдвиг одной строки:
TIME_BUCKETS - 1 = 799 int

сдвиг всех строк:
200 × 799 = 159 800 int

очистка последней колонки:
200 записей нуля

случайные события:
100 попыток добавить volume в последнюю колонку

То есть один tick генерации — это примерно:

159 800 копирований int
200 очисток ячеек
100 случайных обновлений volume
1 repaint request

При таймере в 100 мс это до 10 tick в секунду:

159 800 × 10 = 1 598 000 копирований int / сек
200 × 10     = 2 000 очисток ячеек / сек
100 × 10     = 1 000 случайных событий / сек

Потом приходит отрисовка. Если Swing реально делает около 10 полных перерисовок в секунду, то paintComponent каждый раз проходит всю матрицу:

200 × 800 = 160 000 проверок ячеек / paint
160 000 × 10 = 1 600 000 проверок ячеек / сек

Для каждой непустой ячейки дополнительно выполняется:

  • расчёт яркости через Math.min(255, volume * 4);
  • создание new Color(...);
  • перевод x/y в px/py;
  • расчёт размера прямоугольника через Math.max(1, ...);
  • вызов fillRect(...).

Сколько таких непустых ячеек будет — зависит от случайных данных. В одной новой колонке мы делаем 100 попыток добавить volume в диапазоне примерно 80 ценовых уровней. Часть попаданий склеивается в одну и ту же цену, поэтому обычно это не 100 разных квадратиков, а меньше.

Но порядок понятен: после прогрева истории на экране могут быть уже десятки тысяч непустых ячеек. И каждая из них превращается в цветной прямоугольник.

Итог по первой версии:

на генерацию:
≈ 1.6 млн копирований int / сек
≈ 2 тыс очисток ячеек / сек
≈ 1 тыс случайных событий / сек

на отрисовку:
≈ 1.6 млн проверок ячеек / сек
+ цвет, координаты и fillRect для каждой непустой ячейки

И вот теперь цель совпадает с выводом: первая версия простая и понятная, но даже в ней “одна новая колонка” — это не одна операция. Это сдвиг истории, генерация данных, полный проход по матрице и пачка рисования.

Пока это нормально. Но теперь хотя бы видно, что именно начнёт дорожать, когда мы добавим больше событий.

В следующей части добавим synthetic order book: появятся заявки, сделки, bid/ask и больше событий на каждый tick.
