# Swing Heat Map — рабочая дорожная карта и состояние проекта

Этот файл — единая точка продолжения работы над проектом и публичными материалами.

Он должен одновременно выполнять четыре задачи:

1. Показывать, зачем существует проект.
2. Давать пошаговый план работы.
3. Хранить текущее состояние, результаты экспериментов и принятые решения.
4. Позволять продолжить работу в новом чате без восстановления всей переписки.

Файл живой: план можно менять по мере того, как становится понятнее итоговая история проекта.

---

## 1. Главная идея

Это Java Swing-прототип визуализации рыночных данных, вдохновлённый интерфейсами вроде Bookmap.

Цель проекта — не создать торговый продукт и не повторить Bookmap. Это инженерный эксперимент:

> Насколько далеко можно продвинуть обычный Java Swing в real-time визуализации, где появляются реальные ограничения и какие оптимизации действительно помогают?

Главный принцип:

> Build things to understand them. Measure before optimizing.

Результатом должен стать не только работающий код, но и законченный инженерный кейс:

- воспроизводимая исходная реализация;
- видимая проблема под нагрузкой;
- измерения и гипотезы;
- улучшенная реализация;
- честные выводы;
- README или статья;
- короткое демонстрационное видео.

## 3. Границы первой публикации

Первая публикация не должна рассказывать обо всём проекте.

Рабочая тема:

> Почему первая Swing-реализация heatmap начала тормозить и как я искал причину.

Достаточный сюжет:

1. Показать современный результат.
2. Вернуться к простой ранней реализации.
3. Объяснить её устройство.
4. Увеличить нагрузку и воспроизвести проблему.
5. Сформулировать несколько гипотез.
6. Добавить или показать измерения.
7. Объяснить найденную причину.
8. Показать изменение реализации и результат.
9. Зафиксировать, что помогло, а что оказалось несущественным.

Необязательно для первой публикации:

- идеальная архитектура;
- production-ready код;
- полный обзор всех классов;
- окончательная оптимизация;
- профессиональный монтаж;
- камера;
- объяснение всех этапов развития проекта.

## 4. Оценка времени

Общий безопасный бюджет: **40–50 часов**.

Минимально публикуемый результат можно получить за **20–25 часов**.

Нормальный законченный материал потребует примерно **30–40 часов**.

| Этап | Оценка |
|---|---:|
| Воспроизведение baseline и подбор нагрузки | 4–6 ч |
| Измерения и профилирование | 6–10 ч |
| Черновая воспроизводимая реализация эксперимента | 6–8 ч |
| Сбор выводов и иллюстраций | 3–5 ч |
| README или статья | 5–7 ч |
| Запись и монтаж простого видео | 6–9 ч |
| Ошибки, тупики и переделки | 5–7 ч |

Календарный ориентир:

- 1–2 часа по вечерам: 3–4 недели;
- вечера и выходные: около 2 недель;
- почти полный рабочий день: 5–7 рабочих дней.

Оценки являются ограничителями, а не обещанием. Если этап сильно выходит за бюджет, нужно уменьшать его масштаб или отдельно зафиксировать причину.

Оценка: 1–2 часа.

- блокировки и синхронизация;
- слишком большая работа на EDT;
- частота repaint;
- преобразование модели данных в render state;
- GC;
- сочетание нескольких факторов.

- [ ] Записать гипотезы до просмотра окончательных результатов.

## 7. Правила исследования

1. Сначала зафиксировать вопрос, затем менять код.
2. Менять по возможности один существенный фактор за раз.
3. Все сравнения проводить в одинаковых условиях.
4. Записывать не только удачные, но и отрицательные результаты.
5. Не использовать красивую цифру, если непонятно, как она получена.
6. Не путать корреляцию с причиной.
7. Не оптимизировать только по визуальному ощущению.
8. Не переписывать рабочую реализацию до сохранения baseline.
9. Сразу сохранять скриншоты и записи — позднее состояние кода может измениться.
10. Публикация должна честно описывать ограничения эксперимента.
